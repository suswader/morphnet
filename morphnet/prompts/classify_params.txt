You are classifying HTTP API parameters to determine their role in a web automation workflow.

For each parameter, classify as ONE of:
- **chained**: The value was derived from a prior API response in this workflow. You MUST specify which prior response field it came from by selecting from the provided chain candidates. Only classify as chained when the source field and target parameter have a clear semantic relationship (e.g., a station code from an auto-suggest response flowing into a train search request's station code field).
- **user_intent**: The value was provided by the user as part of the task. This includes search terms the user typed, dates they selected, quantities they chose, or other task-specific inputs. Date formats may differ between the task description and the API (e.g., "25 April 2026" vs "25-04-2026") — still classify as user_intent if the semantic meaning matches.
- **website_generated**: A constant, configuration value, or site-specific default not provided by the user and not derived from a prior response. This is the DEFAULT classification — most parameters are website_generated. Examples: API keys, feature flags, sort orders, language codes, pagination limits, boolean config values, device IDs.

## Classification Priority

1. First check chain candidates: if a candidate has a clear semantic name match with the target parameter (e.g., response field "stationCode" → request param "sourceStationCode"), classify as **chained**.
2. Then check user intent: if the value semantically matches something from the task description or user actions (accounting for format differences), classify as **user_intent**.
3. Everything else is **website_generated**. When in doubt, classify as website_generated.

## Important Rules

- **Most parameters are website_generated.** Boolean values (true/false), numeric configs (limits, counts), API keys, device IDs, feature flags — all website_generated.
- **Not all chain candidates are real chains.** A value like "PUNE" might appear in a popular stations list coincidentally. Only chain when the source endpoint semantically feeds the target endpoint (e.g., auto-suggest → search, not popular-stations-list → search).
- **Chaining requires semantic name relevance.** The source field name should relate to the target param name. "stationCode" → "sourceStationCode" is valid. "majorStn" → "enableNearby" is NOT valid even if both are "true".
- **Prior node context matters.** Use the source node's endpoint path and the CU action that triggered it to judge whether the chain is real.
- **Profile history helps.** If a parameter always has the same value across prior tasks, it's likely website_generated. If it varies per task, it's likely user_intent.
